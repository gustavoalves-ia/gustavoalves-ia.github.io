Projeto carrossel

Como separar fundo, texto e render local deixou o fluxo de Instagram menos mágico e mais confiável.

Pensa numa coisa que parece simples até você tentar fazer direito: gerar um carrossel de Instagram com IA sem deixar a IA escrever texto errado dentro da imagem.

Porque esse é o problema. A gente olha para uma ferramenta de imagem e pensa: "beleza, faz o slide inteiro". Fundo, texto, composição, estética, tudo no mesmo bolo. Parece mais fácil.

Só que o mais fácil, nesse caso, vira o mais frágil.

O modelo de imagem é bom para criar atmosfera. Fundo, textura, objeto, clima. Mas texto? Fonte exata? Hex exato da paleta? Margem segura? Acento em português? Aí começa o b.o.

Não é porque IA não presta não, é porque eu estava pedindo para ela fazer a parte errada.

i. o corte que resolveu separar imagem de composição

A virada foi separar o carrossel em duas camadas.

Uma camada é imagem. Essa fica com o Gemini: capa e fechamento recebem um fundo visual, sem texto, sem letra, sem tentativa de diagramar frase.

A outra camada é composição. Essa fica local, no meu computador: HTML, CSS, fonte Space Grotesk, paleta fechada, Chrome gerando PNG.

Parece menos mágico.

E justamente por isso funciona melhor.

O texto do slide sai do roteiro.md (o arquivo com o roteiro do carrossel). O Chrome renderiza esse texto por cima do fundo. A fonte vem de um arquivo local. A cor vem de constante no código. A margem é número, não sugestão.

O modelo de imagem não tem que "entender" a identidade visual inteira. Ele só precisa fazer um fundo bonito dentro de um estilo.

Pensa. É uma diferença pequena na arquitetura, mas enorme no resultado.

ii. menos IA, mais controle usar modelo só onde ele agrega

Esse projeto começou com uma decisão meio contraintuitiva: usar IA só onde ela agrega.

Não onde ela impressiona.

Eu poderia tentar fazer um prompt gigante: "gere um carrossel em 5 slides, com tipografia tal, paleta tal, texto tal, sem errar acento, sem mudar frase, mantendo margem". Só de escrever isso já dá para sentir o cheiro da fiotagem.

Porque aí qualquer erro vira negociação com o modelo. Regera. Ajusta prompt. Tenta de novo. Acha que foi. O texto veio torto. A fonte não bateu. O fundo ficou legal, mas o slide morreu.

No fluxo novo, a parte que precisa ser determinística fica determinística.

Determinística (ou seja, sem depender do humor do modelo).

O miolo do carrossel nem chama o Gemini. É fundo chapado, alternando claro e escuro. Capa e fecho chamam imagem. Se eu quiser refazer só a capa, rodo com --slide 1. Se quiser revisar tudo, gero a folha-contato.

É mais ferramenta e menos espetáculo.

iii. o bug que entregou a verdade teste pequeno segurando decisão grande

Teve um bug pequeno que mostrou por que teste importa.

No roteiro, um slide pode ter:

imagem:
prompt: uma coluna grega

Ou seja: imagem vazia, mas com um prompt para gerar fundo.

O parser estava lendo errado. Como o campo imagem: estava vazio, ele engolia a linha de baixo e achava que a imagem era literalmente prompt: uma coluna grega.

Resultado: o sistema tentaria abrir isso como caminho de arquivo.

Coisa besta. Coisa que passa fácil quando você só olha o código e pensa "deve funcionar".

Mas no teste apareceu. Escrevi a falha, corrigi o parser, rodei de novo. Verde.

Essa parte é meio sem glamour, mas é onde o projeto fica de pé.

iv. o limite real quando o código chega e a conta barra

O fluxo local funcionou.

Chrome gerou os PNGs. O roteiro virou slides. A folha-contato saiu. A composição ficou controlada.

O que não fechou ainda foi o Gemini real, por quota. Precisa pagar né, uma coisa é você abrir o Gemini mandar o prompt baixar a imagem, outra coisa é você pedir tudo pelo lugar onde esta acostumado a trabalhar, no meu cado o Próprio Obsidian (sim coloquei meus agentes aqui dentro).

A chamada chegou na API. Primeiro tinha um erro de endpoint, eu estava usando v1 onde a geração de imagem precisava de v1beta. Corrigi. Depois a API respondeu de verdade: 429, limite de uso.

Em português claro: o código bateu na porta certa, mas a conta não tinha crédito/quota para entrar.

Isso é um tipo bom de problema. Chato, mas bom. Porque não é "não sei se funciona". É "falta liberar o recurso externo".

Enquanto isso, fiz um teste local com Gemini falso e Chrome real. O sistema gerou os slides em uma pasta temporária. Isso prova o pedaço que é meu: leitura do roteiro, decisão de fundo, escrita dos arquivos, HTML, render, folha-contato.

O pedaço que é do Google depende de billing.

v. o que ficou pronto um comando, um fluxo, uma regra

Ficou um comando:

node ia\mapas\conteudo\instagram\render\src\post-ig.mjs teste-fase-3

Ficou um fluxo:

roteiro.md -> fundos de capa e fecho -> HTML com texto -> PNG -> folha-contato.

Ficou uma regra:

modelo de imagem não escreve texto.

E ficou uma spec para outro modelo continuar sem depender da minha memória da sessão. Isso para mim é importante. Não basta fazer funcionar hoje. Tem que deixar rastro para retomar amanhã.

Porque projeto pequeno também apodrece se ninguém entende onde está a decisão.

vi. a lição menos pedido mágico, mais responsabilidade separada

A primeira versão mental era "vamos gerar carrossel com IA".

A versão que prestou foi outra: "vamos usar IA para a parte incerta e código para a parte que precisa ser exata".

Essa frase vale para muito mais coisa do que carrossel.

Tem uma ansiedade em automatizar tudo com modelo. Só que às vezes a automação boa nasce quando você tira trabalho do modelo.

Menos pedido mágico. Mais separação de responsabilidade.

Quem quer tem que querer.
