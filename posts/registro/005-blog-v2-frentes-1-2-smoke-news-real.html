Errando e melhorando, primeira news real publicada

Duas skills paralelas pro pipeline de conteúdo, dois bugs visuais corrigidos no caminho e uma regra que ficou em três lugares pra não voltar.

Sentei pra fechar duas pendências do blog. Tinha decidido a ordem semana passada, listing primeiro, news depois. Levantei com primeira news no ar, dois bugs corrigidos no caminho e uma regra que ficou em três lugares pra não voltar.

i. o problema da fase 2 um pedaço grudou fora do fluxo

Na fase 2 do refactor do fluxo de conteúdo eu separei a skill que vira .md em HTML (md-html-v2, o motor de renderização do blog) da skill que monta o listing (a página com índice de posts no caderno). Funcionou pra publicar. Mas o listing ficava defasado, ninguém atualizava automático. Eu tinha que editar index.html (o HTML do listing) na mão depois de cada peça.

Pensa. Pra um sistema que se chama "fluxo de conteúdo", deixar um pedaço grudado fora do fluxo é gambiarra.

Frente 1: fazer o listing auto-atualizar quando o HTML nasce.

ii. frente 1, listing sozinho skill paralela que se atualiza quando o post nasce

Brainstorm com quatro perguntas, decisão fechada em cada:

  • → Skill v2 paralela à v1 (não refatora, convive). Mesma estratégia da fase 2, deu certo.
  • → Auto-detect da pasta de drafts (drafts-v2/, onde mora rascunho v2) sem flag.
  • → Chamada automática dentro do md-html-v2 antes do deploy.
  • → Falha do listing não trava deploy. Aviso visível força resolver depois.

Saiu rápido. Skill criada, comando slash criado, passo novo no md-html-v2, anti-padrão "bloquear deploy por causa de listing" anotado.

Smoke test destrutivo (rerodar a skill no único post v2 que existia) eu pulei. O HTML do listing tinha um realce de cor que eu tinha feito na mão na fase 2. Rerodar apagaria. Deixei pra validar no primeiro post v2 nascendo do zero.

iii. frente 2, news vira fonte cada notícia vira matéria-prima pra N canais

Aqui o brainstorm foi mais longo. News tem estrutura rica (cronologia, ticker que roda no topo, timeline lateral, citação de fontes secundárias, imagem hero). E tem um princípio do roadmap que eu queria respeitar: separar fonte de canal.

Princípio: cada peça nasce uma vez como fonte neutra, depois N canais (blog, IG, LinkedIn, TikTok) reapuram a partir dela.

Decisão: a skill post-news-v2 produz dois arquivos numa execução. Um cru reusável (em sessoes/news/<slug>.md, fora do hub do blog), com a notícia raspada mais a minha posição em três perguntas-chave (testou? primeira coisa que mudou? veredito?). E um draft de blog pronto (em drafts-v2/news/<slug>.md), que aponta de volta pro cru via campo origem: no frontmatter (o cabeçalho de metadados no topo do arquivo).

Quando eu quiser virar a notícia em carrossel de Instagram amanhã, a skill de IG (que ainda não existe) consome o cru, não o blog. Cada canal reapura.

Outra decisão grande: posts compósitos. Quando rolam três notícias relacionadas no mesmo dia, eu passo as três URLs, a skill gera três crus atômicos (cada um reusável solo pra IG) e um draft de blog que combina os três via origem: [lista].

O n-stamp (cada atualização cronológica no post de blog) virou heading Markdown ## 14:08 título · fonte em vez de array YAML aninhado no frontmatter. Edita melhor no Obsidian (editor de Markdown que uso), lê melhor visualmente.

iv. smoke híbrido sintético primeiro, real depois

Antes de testar com URL real, propus dois níveis. Sintético primeiro, pra pegar bug estrutural barato. Real depois, pra pegar voz, UX e bug visual.

Sintético: criei fixtures (drafts e crus falsos com prefixo teste-), rodei a lógica das skills, validei mutações nos arquivos, restaurei tudo via snapshots em /tmp (pasta temporária do sistema). Cleanup garantido. Três tests passaram (parser de n-stamp, normalização de origem: como lista, atualizar-listing-v2 com peça nova).

Real: peguei landing de news da Anthropic, escolhi Opus 4.8 que tinha saído de manhã. Aí o primeiro travamento.

A ferramenta que raspa página (Firecrawl) não tava instalada nesse ambiente. Caí pro WebFetch como alternativa, raspei na unha.

Anthropic não expõe hora exata da publicação no metadata. Marquei hora: ~13:00 BRT (aproximada) com nota. Inventar hora exata era falsidade, melhor admitir aproximação.

Três perguntas-chave: não testei ainda (saiu mesmo dia que vi), a segunda eu pulei (sem teste não tem o que dizer), veredito "preço continua bom mas sem teste real fica leitura". Cru e draft salvos.

v. render e dois bugs a spec não previu, o real mostrou

Como o pipeline ainda não tem agente autônomo que execute a skill md-html-v2 por mim, gerei o HTML manualmente seguindo o COMPOR (o template de como cada categoria do blog vira HTML). News é grande, uns 210 linhas com ticker, timeline, hero, corpo com dois n-stamps. Setup de estilos primeira vez no caderno: copiei post.css, news.css e fontes (JetBrains Mono) de uma pasta de referência.

Aí veio bug número 1.

Abri o HTML local, hero não apareceu. Path estava com três níveis .. (subir diretório), precisava ser cinco. HTML mora em blog/posts/news/, hero mora em ia/sessoes/news/<slug>/. A diferença é cinco diretórios pra cima e dois pra baixo. Eu tinha escrito três na spec sem testar. Corrigi em quatro lugares de uma vez (HTML, draft, skill, spec). Pra próximo uso sair certo de primeira.

Bug número 2 foi mais sério em cara.

Refresh. "Totalmente desconfigurada, sem adequação com a página". Pensei "ferrou, CSS quebrou".

Não era CSS. Era a hero. A imagem original da Anthropic era 2880×1620. Sem class no <img>, ela renderiza tamanho real. Atropela a grade de três colunas do post (TOC à esquerda, conteúdo no meio, timeline à direita). A coluna do meio, que devia ter uns 700 pixels, virou 2880 pixels. Tudo desalinhado.

O CSS tinha regra .t-fig-img { width: 100%; height: auto }. Mas eu não tinha colocado a class no <img>. Fix simples, lição cara.

Anotei a regra em três lugares pra não esquecer:

  • → Anti-padrão no SKILL.md da skill que renderiza
  • → Mapping atualizado no COMPOR de news (que documenta o que cada campo vira em HTML)
  • → Memória persistente do agente, com Why e How to apply explícitos

Se rolar de novo em qualquer categoria, o agente vai capturar antes de eu abrir o arquivo.

vi. deploy três coisas a resolver antes do push

Comando /deploy-blog. Antes do push, três coisas no git status (a foto do que mudou):

  • → Skills v2 untracked (deveriam estar no .gitignore, esqueci de adicionar quando criei). Adicionei.
  • → Hero path saindo do repo. GitHub Pages serve só uma pasta como raiz. O cru com hero mora fora dessa pasta. Tive que copiar a imagem pra dentro do repo (blog/assets/news/<slug>/hero.jpg) e ajustar path do HTML pra local. Pendente: isso devia rolar automático na skill md-html-v2. Anotei.
  • → Um post de teste antigo apareceu no status. Confirmei com migo mesmo (era teste), deixei de fora.

Commit post 004 news · Opus 4.8 sai, preço fica, push. SHA 25c1d60. A primeira peça do caderno news no ar em https://gustavoalves-ia.github.io/posts/news/004-claude-opus-4-8.html.

Que cara loko ver listing nascer com boletim · 1 peça.

vii. o que ficou duas skills, três regras documentadas

Duas skills v2 entregues e em uso. Uma news real publicada. Dois bugs corrigidos em mais de um lugar (spec, skill, memória) pra não voltar. Uma regra do design system (<img class="t-fig-img">) virou anti-padrão documentado.

Bug semântico pequeno no md-html-v2 ficou na lista. A regra "marca cru como processado só se a lista de derivados era vazia" quebra quando a skill post-news-v2 já popula a lista com draft: antes do render. Mudei manual nesta peça. Próxima sessão ajusto a regra pra "marca como processado quando este HTML for o primeiro blog: na lista".

Frente 3 (promover v2 pra default, renomear v1 pra .bak) fica pra quando rodar mais peças.
